Chess-Aware Position Encodings for Attention

The problem: Your current transformer uses learned absolute position embeddings — a 64-entry lookup table added to token embeddings at input. This tells the model "this square is e4" but nothing about how squares relate. The model has to learn from scratch that e4 and d5 are adjacent, that a1 and h8 are on a diagonal, that a knight on c3 attacks e4.

Both approaches below inject geometry directly into the attention logits — the scores that decide which squares attend to which — so the model doesn't have to discover chess topology on its own.

1. Lc0's 64×64 Absolute-Pair Bias

The idea

For every ordered pair of squares (i, j), learn a scalar bias b[h, i, j] per attention head. Add it to the attention logit before softmax. That's it.

attn_logit[i,j] = (q_i · k_j) / √d  +  bias[head, i, j]
attn_weight = softmax(attn_logit)

It's a full (H, 64, 64) lookup table. Since every square-pair gets its own parameter, the model can learn any spatial relationship — diagonals, knight jumps, "back rank", "same file", "this exact pair" — without any structure imposed.

Shape & params

At H=8 heads: 8 × 64 × 64 = 32,768 params per layer. At L=8 layers: 262K total (~4% of a 7M model). Trivially small.

Gotcha: passing attn_mask may disable the flash-attention kernel. At T=64 the math-mode fallback is cheap (~4% of FLOPs), but measure throughput to be sure.

2. Shaw-style 2D Relative Encoding (ChessFormer)

The idea

Instead of one bias per absolute pair, learn one per displacement. Two squares relate by a vector (Δrank, Δfile), each ranging from -7 to +7. That's 15 × 15 = 225 possible displacements.

The key insight: chess is mostly translation-invariant. A knight on c3 attacking e4 is (Δr=+1, Δf=+2). A knight on f6 attacking h7 is also (Δr=+1, Δf=+2). Same displacement → same attention bias. The 64×64 approach would learn these as two separate parameters; Shaw forces them to share.

Knight on c3 → e4

Δrank=+1, Δfile=+2

Knight on f6 → h7

Δrank=+1, Δfile=+2 — same bucket

The 15×15 displacement table

Each cell is one bucket. The center (0,0) is "same square". The 8 green cells are the knight displacements — any knight move, anywhere on the board, hits one of these 8 buckets.

Center (black) = Δ=(0,0). Green = the 8 knight-move displacements.
A bishop-specialized head would light up the diagonals through center; a rook head would light up row 0 and column 0.

How it plugs into attention

Shaw's original formulation learns vectors per bucket (not scalars), and they participate inside the dot product rather than as an additive bias:

e_ij = ( (x_i W_Q + a_Q[Δ(i,j)]) · (x_j W_K + a_K[Δ(i,j)]) ) / √d

where a_Q, a_K ∈ ℝ^(225 × d_head) are lookup tables indexed by the displacement bucket. This is more expressive than a scalar bias — the relative-position vector interacts with content via the dot product. But it means you can't use the fused SDPA kernel; you hand-roll attention.

A simpler "scalar bias" variant (also used in some papers) learns just bias[H, 225] and adds it to logits exactly like the 64×64 approach — but with 225 params/head instead of 4096. Less expressive, but drops into SDPA the same way.

Shape & params

Full Shaw: 225 × d_head × 2 (for a_Q and a_K), shared across layers. At d_head=32: ~14K total regardless of L or H. Tiny.

Code sketch

# Precompute at init: (64, 64) tensor mapping each (i,j) pair to its displacement bucket 0..224
rank_i, rank_j = sq_i // 8, sq_j // 8
file_i, file_j = sq_i %  8, sq_j %  8
bucket = (rank_j - rank_i + 7) * 15 + (file_j - file_i + 7)  # shape (64, 64), values in [0, 225)

# Learnable tables (shared across layers)
self.rel_q = nn.Parameter(torch.zeros(225, d_head))
self.rel_k = nn.Parameter(torch.zeros(225, d_head))

# Forward: gather then hand-roll attention
aq = self.rel_q[bucket]  # (64, 64, d_head)
ak = self.rel_k[bucket]
logits = einsum("bhid,bhjd->bhij", q, k)
logits += einsum("bhid,ijd->bhij", q, ak)         # content-position
logits += einsum("ijd,bhjd->bhij", aq, k)         # position-content
logits += einsum("ijd,ijd->ij", aq, ak)[None,None] # position-position
attn = softmax(logits / sqrt(d_head))

Which to try first?

Start with 64×64 — it's 3 lines, keeps your existing SDPA path, and Lc0's result is strong. If it works, you've gotten most of the benefit. Then optionally layer Shaw on top or swap to it for the extra ~1pp.

The interesting architectural question is whether 64×64's extra expressiveness (absolute-pair patterns) actually helps, or whether Shaw's forced weight-sharing generalizes better on your 342M positions. With that much data you're probably not overfitting the 262K bias params, so 64×64's extra capacity is likely pure upside — but ChessFormer's ablation says otherwise. Worth testing both.
